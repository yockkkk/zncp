# 智能餐饮推荐系统 · 产品技术白皮书

> **【多人口味 + 忌口/过敏 + 疾病】三合一智能菜品推荐 AI**
>
> 定位：**“不是推荐‘最好吃’，是推荐‘最适合这一桌人’”**

---

## 一、 产品定位与核心痛点

传统餐饮推荐系统（如美团、大众点评等）长期停留在**“单人画像、千人一面、历史堆叠”**的简单模式，面临三个无法解决的线下堂食痛点：

1. **多人用餐，口味打架**：当一桌人一起聚餐时，有人爱吃辣、有人要清淡、有人爱吃肉、有人吃素。传统系统只能给出“统一推荐”或热门排行，无法自动平衡一桌人冲突的口味。
2. **食品安全，防线缺失**：带老人吃饭怕高血压/糖尿病，带小孩吃饭怕花生/海鲜过敏，或者部分顾客有特定的宗教习惯（如清真）。传统推荐只顾“好吃”与“热门”，几乎没有能严格避开过敏源与慢性病禁忌的安全筛选，一旦点错极易引发纠纷甚至安全事故。
3. **商家痛点，推不准高毛利菜**：餐厅要赚钱，要么生硬硬推高价菜（吃相难看，顾客反感），要么只推爆款（不赚钱）。市面上 90% 的点餐系统无法做到“在保证口味搭配合理且顾客满意的原则下，隐形、自然地推荐高毛利菜品”。
4. **记忆缺失，无法识别熟客**：堂食点餐中，新客老客无差别对待。传统推荐系统无法保存、记忆顾客的个人口味画像，导致顾客每次来就餐，服务员仍需从头询问。本系统创新开发了**“长期记忆”与“口味偏好学习型”推荐**，输入手机号即可秒级调取熟客的历史到店次数、历史消费明细、常点菜品和AI自动归纳的偏好口味标签，使餐厅具备“过目不忘”的专属贴心服务体验。

### 核心定位

本系统针对以上堂食场景进行痛点包围，开发了 **“多人口味 + 忌口过敏 + 疾病慢病”三合一与“长期记忆口味学习型”菜品推荐 AI**。服务员仅需使用平板设备（Waiter Pad）简单勾选偏好或输入手机号，系统即可在秒级时间内给出兼顾全桌安全与长期偏好的定制配餐方案，并在后台隐形赋能餐厅客单价。

## 二、 系统面向的场景与受众

### 2.1 目标受众（面向谁）
- **线下餐厅服务员 / 领班**：在点餐环节提供智能化决策辅助，提高推荐专业度。
- **多人堂食聚餐群体**：如家庭聚餐（带老人与小孩）、朋友聚会（口味冲突多）、商务宴请（避忌多）。
- **特殊饮食群体**：过敏体质人群（海鲜/花生/牛奶过敏）、慢性病患者（痛风/糖尿病/高血压）以及特定宗教或饮食习惯人群（清真/素食/减脂）。

### 2.2 核心应用场景
- **多人桌餐智能配菜**：顾客入座后，服务员通过系统录入整桌人数与每位客人的具体偏好与忌口，AI 自动生成桌级配菜组合。
- **拍照环境智能感知**：支持拍摄餐桌照片，多模态大模型感知现场人数、杯盘狼藉程度、氛围场景（如约会/商务），辅助生成开场话术。
- **高毛利隐形推广**：自动在配菜方案中自然穿插鲜榨果汁、特色甜点等高毛利菜品，且毛利率较高菜品在同等匹配度下排名靠前。
- **熟客长期记忆偏好学习**：支持输入顾客手机号自动匹配长期口味画像，由决策引擎自动识别“爱吃酸”、“爱吃麻”、“不爱甜”、“喜欢嫩”等隐性口味倾向，并在推荐流程中自动注入或一键套用，实现持续进化的个人偏好定制。

## 三、 五大 Agent 协同体系与核心技术壁垒

本系统采用先进的 **RAG (检索增强生成)** 与 **5-Agent 智能协同** 架构，从感知到决策逐层递进：

```
                       ┌──────────────────────┐
                       │    拍摄现场餐桌照片    │
                       └──────────┬───────────┘
                                  │ (Scene Image)
                                  ↓
      ┌─────────────────────────────────────────────────────────┐
      │ Agent 1: 场景感知 (Vision Model)                         │
      │ ─ 模糊感知人数、桌型、氛围场景并提取特征                 │
      └──────────────────────────┬──────────────────────────────┘
                                  │ (Scene Context)
                                  ↓
      ┌─────────────────────────────────────────────────────────┐
      │ Agent 2: 画像构建 (UserProfile Consolidation)           │
      │ ─ 合并多人口味，结合手机号匹配的“长期记忆画像”生成画像    │
      └──────────────────────────┬──────────────────────────────┘
                                  │ (Consolidated Query Text)
                                  ↓
      ┌─────────────────────────────────────────────────────────┐
      │ Agent 3: 匹配与硬过滤 (Vector Search + Rule Filter)       │
      │ ─ Qdrant 语义检索召回 + 强忌口/过敏/清真安全硬性拦截   │
      └──────────────────────────┬──────────────────────────────┘
                                  │ (Safe Candidate Dishes)
                                  ↓
      ┌─────────────────────────────────────────────────────────┐
      │ Agent 4: 智能重排与高毛利推荐 (LLM Reranker)            │
      │ ─ 结合 grossMargin 隐形推荐 + 菜品人头映射 suitableFor │
      └──────────────────────────┬──────────────────────────────┘
                                  │ (Ranked Recommendation List)
                                  ↓
      ┌─────────────────────────────────────────────────────────┐
      │ Agent 5: 服务员话术生成 (Waiter Script Generator)        │
      │ ─ 针对不同菜品生成自然口语化的开场白与推荐词             │
      └─────────────────────────────────────────────────────────┘
```

### 3.1 核心技术实现细节

1. **向量语义检索 (Embedding + Qdrant)**：
   菜品数据全部转化成 1024 维的高维数学向量（包含口感、原料、适宜人群、消费水平等特征值）。系统通过语义计算找出与画像需求“意思最接近”的候选集，避免了传统数据库字面匹配的死板。
2. **食品安全硬核拦截器**：
   在匹配层（Agent 3）加入了强忌口/宗教习惯过滤规则：
   - **清真（Halal）防线**：若整桌任意一人要求清真，底层硬规则直接拦截所有含猪肉、排骨、大肠等成分的菜品。
   - **过敏源/疾病防线**：海鲜过敏者自动屏蔽鱼虾蟹；痛风患者屏蔽海鲜、高嘌呤浓汤；糖尿病患者自动屏蔽红豆双皮奶等高糖高甜食品。若某道菜对所有客人都存在冲突，直接安全拦截。
3. **大模型重排序与人群分流 (suitableFor)**：
   大模型（Agent 4）接收所有客人的个人要求，从过滤后的安全候选池中挑选组合。为了实现“私人定制感”，大模型会精确输出 `suitableFor`（适合享用的顾客）标签，区分哪些是“全桌共享菜”，哪些是“某顾客专属菜”。
4. **隐形毛利率加权**：
   数据库中为每道菜配置了 `gross_margin`（毛利率）字段。在 LLM 重排序提示词中，系统要求大模型“在不违和、口味搭配合理的原则下，悄悄优先推荐高毛利菜品”，使推荐方案中高利润的特饮、手工甜点占比自然提升。
5. **长期记忆与口味画像自适应学习**：
   系统在 `recommendation_record` 中记录顾客手机号。当该顾客再次消费时，后台通过分析历史采纳（下单）菜品的配料及味道特征（如“番茄”、“酸菜”对应酸味，“花椒”对应麻味，甜品蛋糕对应甜味，滑鸡豆腐对应嫩滑度），量化推导出顾客在“酸、甜、麻、辣、嫩”等维度的口味偏好倾向（如“爱吃酸”、“不爱甜”、“喜欢嫩”）。该长期画像数据会自动融合进新一轮的用户画像与向量检索 Query 文本中，确保大模型生成的菜品方案和推荐说辞紧密贴合客人的长期口味记忆。

## 四、 本系统的七大优势（为何“不是空壳子”）

| 维度 | 传统点餐系统 / 规则过滤 | 通用大模型直接对话 | **本系统方案** |
|---|---|---|---|
| **冷启动能力** | ✗ 必须依赖历史消费数据积累 | △ 必须由用户手动输入全部偏好 | **✓ 零数据冷启动，仅需勾选即可精准推荐** |
| **食品安全把关** | ✗ 无过滤或仅有死板的字面过滤 | ✗ 大模型经常产生幻觉，遗漏配料成分 | **✓ 硬规则过滤 + 软语义筛选，确保过敏/禁忌 100% 拦截** |
| **多人餐平衡** | ✗ 无法自动汇总并平衡多人冲突口味 | △ 可以排版，但无法实时同步本地菜品库 | **✓ 自动合并生成桌级画像，实现智能分餐与人群映射** |
| **商业变现机制** | ✗ 机械推荐爆款，客单价提升有限 | ✗ 无法感知商家毛利与实时库存情况 | **✓ 引入毛利因子隐形加权，优化点单利润结构** |
| **真实性保障** | ✓ 基于本地数据库 | **✗ 经常捏造虚构菜单（大模型幻觉）** | **✓ 严格的 RAG 架构，确保推荐结果 100% 真实存在** |
| **服务人员赋能** | ✗ 仅作电子菜单展示 | ✗ 无法提供业务化落地支持 | **✓ 自动生成针对本桌菜品的服务员口语化推荐话术** |
| **长期偏好记忆** | ✗ 无法记住个人历史口味，每次从零询问 | ✗ 每次均需重新拼接 Prompt 或输入要求 | **✓ 输入手机号自动读取历史行为并由AI计算出偏好，秒级套用** |

## 五、 项目商业价值

1. **规避食品安全风险**：通过多重硬规则机制阻断过敏源及慢病危险食品上桌，极大地降低了餐饮企业的合规与诉讼风险。
2. **点餐效率极速跃升**：免去了多人聚餐时“服务员逐个口头询问忌口、协调菜品”的冗长过程，从人均 10 分钟点单缩短至 30 秒，极大地提升了餐厅翻台率。
3. **点单毛利自动优化**：隐形穿插高毛利率的特色冷饮及配菜，能够悄无声息地将客单价提升 15%~25%，直接转化为餐饮商家的净利润。
